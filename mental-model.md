## Behavioral Patterns

### Decision-Making Style
- **Concepts and systems first, then analogies** (Confidence: 95% — validated)
  - Quote: "I tend to think in concepts and systems and then I like to use analogies to stress test or to explain"
  - Implication: I should present frameworks/structures. Analogies come FROM Will to validate, not from me to explain.
  - Validated: Multiple sessions, including junior-senior analogy in March 7

### Thinking Process
- **Two-phase approach** (Confidence: 90%)
  - Phase 1: Build conceptual/systems understanding (exploration mode)
  - Phase 2: Use analogies and specifics to stress-test or communicate (validation mode)
  - Then → Implementation

### Mode Patterns

**Confirmed sequence**: Exploration → Summary/Confirmation → Implementation (Confidence: 95% — multi-session validated)
- Signals of exploration: Abstract language, "what if", comparisons, framework questions
- Signals of confirmation: Analogies, summary statements ("I think this fits"), "let's call this..."
- Signals of implementation: File names, specific actions, "let's implement X", "can you do..."
- **Critical rule**: Do NOT jump to implementation during exploration, even with concrete-sounding ideas

**Mode detection examples**:
- "I want to brainstorm on the what framework first. Rather than planning" → exploration, stop
- "I think this fits. Let's call this new model..." → confirmation, summarize and document
- "Let's now start to take this framework and think about how it will apply..." → still conceptual, resist implementation

**Ambiguity protocol**: When mode is unclear, ask one short question rather than guessing.

### Problem-Solving Style
- Prefers simple, known-good solutions over clever-but-risky ones
- Evidence: Reverted typing indicator to original after debounced approach had a flaw
- "When in doubt, revert to what works" mentality
